System Requirements

Assistance Form Requirements

ID: SR01 – The Assistance Module will provide a simple form for user/families to submit information and requests

ID: SR02 – The Assistance Module will allow staff to view and manage requests

ID: SR03 – The Assistance module will allow families to upload optional documents with the request

ID: SR04 - The Assistance module will let families track request status (submitted/approved/scheduled/fulfilled)

Volunteer Registration Requirements

ID: SR05 – The Volunteer module will show a calendar/list of events and shifts

ID: SR06 – The Volunteer module will let volunteers claim or cancel shifts

ID: SR07 – The Volunteer module will support waitlists and auto-fill open spots

ID: SR08 – The Volunteer module will send signup confirmations and provide add-to-calendar links

Events Calendar Requirements

ID: SR09 – The Events module will display upcoming distributions and outreach events

ID: SR10 – The Events module will show event detail pages with time, place, and directions

ID: SR11 – The Events module will let users subscribe to event updates

Impact Dashboard Requirements

ID: SR12 – The Impact module will provide staff with visual Impact data (meals, families, hours, donations)

ID: SR13 – The Impact module will display public impact counters on the website

ID: SR14 – The Impact module will allow admin edits and exports

Blog/Stories Requirements

ID: SR15 – The Stories module will let editors publish stories and updates

ID: SR16 – The Stories module will allow testimonials to be viewed

ID: SR17 – The Stories module will allow community members and volunteers to comment

Contact Page Requirements

ID: SR18 – The Contact module will list program emails, phone numbers

ID: SR19 – The Contact module will route form inquiries to the correct program/admin inbox

ID: SR20 – The Contact module will send an automatic acknowledgement with a reference number

Staff Dashboard Requirements

ID: SR21 – The Staff Dashboard will provide a unified view of requests, volunteers, and donations

ID: SR22 – The Staff Dashboard will let staff assign items and add internal notes

ID: SR23 – The Staff Dashboard will support bulk actions (approve, message, export, status updates)

ID: SR24 – The Staff Dashboard will have required authentication to access

QR Code Requirements

ID: SR25 – The QR module will generate unique QR codes for each form

ID: SR26 – The QR module will scan codes to validate attendees and record check-in/out

Automated Notifications Requirements

ID: SR27 – The Notifications module will send status updates for assistance requests

ID: SR28 – The Notifications module will send reminders and changes for volunteer shifts

ID: SR29 – The Notifications module will alert subscribers to event changes or cancellations

Multilingual Support Requirements

ID: SR30 – The Language module will allow users to view webpage content in Spanish

ID: SR31 – The Language module will allow request forms to be viewed in Spanish

FAQ Page Requirements

ID: SR32 – The FAQ module will let editors manage FAQ categories and Q&A entries

ID: SR33 – The FAQ module will let users rate answers and suggest new questions

Use Case Descriptions

 

Use Case Name: Submit Assistance Request

ID: 1

Importance Level: High

Primary Actor: Family

Use Case Type: Detail, Essential

Stakeholders and Interests:

 

-    Family: wants to seek help through the Hope Buss system and submit a request form

-    Administrator:

Brief Description: A family completes the simple form with household info and requested resources and submits it.

Trigger: The user clicks the “Submit” button on the assistance request form

Type: External

Relationships:

Association: Family

Extend: Upload Supporting Documents

Normal Flow of Events:

The Family opens the Request Assistance form on the Hope Buss website or mobile interface.

The System displays all required fields for household information and requested resources.

The Family enters all required data household details, contact info, and requested resources.

The Family optionally uploads supporting documents.

The System performs input validation for completeness and correct formats.

The Family clicks the Submit button.

The System saves the request in the database and assigns a unique reference number.

The System sends a confirmation message to the Family.

The System displays a confirmation page with the reference number and a link to track status.

SubFlows:

Alternate/Exceptional Flows:

1a. If the family clicks Submit multiple times, it will detect a duplicate and prevents other clicks

5.     Staff updates the request’s status, assigns it to a team member, or adds internal notes.

6.     The system validates permissions and saves the updates.

7.     If a status change occurs, the system triggers the notification process.

8.     The system logs the modification in the audit trail.

9.     The updated record appears in the queue with revised status and assignment.

Use Case Diagram

volunteers open the system and looks through events and shifts. When they find one that works, they claim the shift (or cancel if plans change). If the shift is already full, they join a waitlist and the system will fill spots as people cancel. After a signup or change, the system sends a confirmation with an add-to-calendar link, so the volunteer can save it right away.
