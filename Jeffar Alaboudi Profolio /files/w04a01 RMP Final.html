1. Introduction

 

This Requirements Management Plan (RMP) defines the process used to identify, analyze, document, track, and control requirements for The Hope Buss Revamped Website Project. The purpose of this project is to modernize and enhance The Hope Buss website to better support community outreach, volunteer coordination, donations, and information sharing.

The Hope Buss system is primarily a web-based software solution consisting of multiple integrated modules including event management, volunteer management, assistance request processing, content management, donations, and analytics. This plan ensures that all functional and non-functional requirements are clearly defined, traceable, and managed throughout the project lifecycle.

 

2. Requirements Management Approach

 

The Derby City Developers project group will follow a structured requirements management approach aligned with PMBOK best practices. This approach ensures that stakeholder needs are accurately captured and translated into system requirements that support The Hope Buss mission.

The requirements management approach consists of four phases:

 

2.1 Requirements Identification

Requirements were gathered through review of organizational goals, vision documents, stakeholder input, and analysis of existing website limitations. Stakeholders include The Hope Buss leadership, administrators, volunteers, donors, and community members.

Primary techniques used include:

  • Stakeholder interviews and discussions
  • Review of existing system documentation
  • Analysis of similar nonprofit web platforms
  • Functional decomposition of proposed features

 

2.2 Requirements Analysis

Identified requirements were analyzed to ensure they were:

  • Clearly defined and achievable
  • Classified as functional or non-functional
  • Categorized as software, web module, database, or integration requirements
  • Assigned a priority level (High, Medium, Low)

Requirements were also evaluated for redundancy and consolidated where necessary to create a manageable and testable requirements baseline.

 

2.3 Requirements Documentation

All approved requirements are documented in the System Requirements Specification and captured within the Requirements Traceability Matrix (RTM). Each requirement is assigned:

  • A unique identifier
  • A hierarchical associated ID
  • A category and priority
  • A status for lifecycle tracking

The RTM serves as the authoritative source for tracking requirement implementation, testing, and validation.

 

2.4 Ongoing Requirements Management

Requirements will be continuously monitored throughout development. Any changes to requirements must follow the configuration and change control process defined in this plan. Requirement status will be updated regularly to reflect progress from Proposed to Approved, Implemented, and Tested.

 

 

3. Configuration Management

Effective configuration management is essential to controlling changes to project requirements and ensuring alignment with project scope, schedule, and budget. The Hope Buss project will follow a formal change control process.

 

The project manager will be responsible for communicating the change to all project stakeholders.

 

3.1 Documentation and Version Control

  • All requirements documentation will be stored in a project repository.
  • Version control will be applied to all requirement-related artifacts.

3.2 Change Control Process

  • Any proposed requirement changes will be brought to attention of the group to be voted upon for approval.
  • Approved changes will be incorporated into the RTM and communicated to all stakeholders.

 

 

4. Requirements Prioritization Process

Requirements prioritization ensures that critical system functionality is delivered while balancing time and resource constraints.

The Hope Buss project uses a three-tier priority model:

  • High Priority: Core system functionality required for operational and user success (e.g., donations, volunteer sign-up, assistance requests).
  • Medium Priority: Enhancements that improve usability and engagement.
  • Low Priority: Features that provide added value but are not essential for initial deployment.

Priorities were established collaboratively by the Derby City Developers team and project stakeholders. As project constraints evolve, priorities may be reassessed through formal review.

 

5. Product Metrics

Product metrics will be used to evaluate system quality, performance, and success. Metrics include:

  • System availability and uptime
  • Page load performance
  • Successful form submissions and transactions
  • Accuracy of volunteer and assistance request tracking
  • User engagement with content and events

Metrics are tied directly to functional and non-functional requirements and will be validated during system testing and user acceptance.
