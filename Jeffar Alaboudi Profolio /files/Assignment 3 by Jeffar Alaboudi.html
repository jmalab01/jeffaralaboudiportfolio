1. Customer Clicks External Links (Use Cases 12 – 13)

 

1.1 Use Case Descriptions:

 

1.1.A. Use Case 12: View Internal Impact Data

 

Use Case Name: View Internal Impact Data

ID: 12

Importance Level: Low

Primary Actor: Staff

Use Case Type: Detail, Essential

Stakeholders and Interests:

 

-    Staff: see up-to-date metrics (meals, families, volunteer hours, donations).

-    Administrator: operational oversight and reporting.

 

Brief Description: Staff open a secure dashboard that displays The Hope Buss’s internal impact metrics such as meals served, families helped, volunteer hours, and donations with filters for date range and program

Trigger: Staff selects Impact → Internal Dashboard after logging in.

Type: External

Relationships:

Association: Staff/ Administrator

Include:

Extend: Use Case 12: Edit and export data

Generalization:

Normal Flow of Events:

 

1.     Staff selects Impact → Internal Dashboard.

2.     System authenticates role and loads default period (last 30 days).

3.     System queries metrics and renders cards + charts.

4.     Staff applies filters (date/program/location).

5.     System refreshes result and allow CSV export button (disabled for read-only roles).

 

SubFlows:

Alternate/Exceptional Flows:

 

A1. No data for range → system shows “No records found” with range quick links.
E1. Permission denied → show 403 and link to request access.

 

 

1.1.B. Use Case 13: View Public Impact Counters

 

Use Case Name: View Public Impact Counters

ID: 13

Importance Level: Low

Primary Actor: Vistor

Use Case Type: Detail, Essential

Stakeholders and Interests:

 

-    Vistor: wants to see real, up-to-date public impact numbers (families served, meals provided, volunteer hours, donations) to understand The Hope Buss’s community reach.

-    Administrator: wants to ensure that the counters display accurate, automatically updated data that builds public trust.

-    Community Partner: wants to confirm their contributions are reflected in the public results and to measure ongoing impact

 

Brief Description: Public page shows rolling counters (families served, meals, volunteer hours, partners).

Trigger: Vistor opens/impact

Type: External

Relationships:

Association: Customer

Include:

Extend:

Generalization:

Normal Flow of Events:

 

1.     Actor opens /impact.

2.     System loads cached counters and last-updated timestamp.

3.     Actor optionally clicks “Learn more” to read methodology.

 

SubFlows:

Alternate/Exceptional Flows:

A1. Cache stale → system refreshes from store and re-renders

1.2.A System Requirements

·       FR012: The system shall display internal impact data (meals served, families reached, volunteer hours, donations) for authorized staff.

·       FR013: The system shall allow staff to filter internal impact data by date range, program, or location.

·       FR014: The system shall show public impact counters that summarize key community metrics on the website.

·       FR015: The system shall automatically update public impact counters when new internal data is entered.

1.2.B Traceability Matrix

 


1.3 Use Case Diagram

 

 

On the Hope Buss website, staff and community visitors can view impact information in different ways. The staff can log in to see detailed internal data, such as meals served, families reached, and volunteer hours. Visitors see public counters showing that same information but in a very simple, visual manner. The "View Internal Impact Data" use case extends the "View Public Impact Counters" use case since both rely on the same data, though the staff have deeper access to detailed reports while the visitors only see summarized numbers.

2.1 Use Case Descriptions:

 

2.1.A. Use Case 14: Edit & Export Impact Data

 

Use Case Name: Edit & Export Impact Data

ID: 14

Importance Level: High

Primary Actor: Administrator/staff

Use Case Type: Detail, Essential

Stakeholders and Interests:

 

Staff Member: wants to be able to update and correct internal impact data such as meals served, families reached, and volunteer hours.

Administrator: wants to ensure data entries are accurate, properly formatted, and securely stored for reporting.

Executive Director: wants to easily export reports and share reliable data with donors, partners, and grant organizations

Brief Description: Admin updates impact entries (e.g., monthly totals) and exports CSV/XLSX for grants/reports.

Trigger: Administrator clicks on “Add a New Trip” on the trip listings page

Type: External

Relationships:

Association: Administrator, Staff Member

Include: UC12 – View Internal Impact Data (since editing/exporting requires first viewing the data)

Extend: UC15 – Publish Stories & Updates (exported data may be used for public updates or reports)

Generalization:

Normal Flow of Events:

 

1.     Administrator logs into the Hope Buss dashboard.

2.     Administrator navigates to the Impact Data section.

3.     System displays internal metrics such as meals served, families reached, volunteer hours, and donations.

4.     Administrator selects “Edit Data” to update numbers or correct entries.

5.     System validates input and saves updated data to the database.

6.     Administrator clicks “Export Report” to download updated data as a CSV or PDF file.

7.     Triggers Use Case 12: View Internal Impact Data

 

SubFlows:

Alternate/Exceptional Flows:

1. Validation fails → inline errors; nothing saved.
2. Export permission missing → disable/export denied.
3. Bulk import → show preview diff before commit.

2.1.B. Use Case 15: Publish Stories & Updates

 

Use Case Name: Publish Stories & Updates

ID: 15

Importance Level: High

Primary Actor: Editor

Use Case Type: Detail, Essential

Stakeholders and Interests:

 

-    Customer: does not want to see trips that are no longer offered

-    Administrator: wants to ensure that trips that are no longer offered are removed

-    Guide: wants to ensure that customers cannot view or book trips that are no longer offered

 

Brief Description: Editor creates/edits/publishes stories, donor spotlights, and program updates.

Trigger: Administrator or Editor clicks on “Publish New Story” or “Post Update” on the Hope Buss dashboard.

Type: External

Relationships:

Association: Administrator, Editor

Include: UC14 – Edit & Export Impact Data (published stories often include updated impact data or reports)

Extend: UC16 – View Testimonials (stories may include or reference community testimonials)

Generalization:

1.     Administrator or Editor logs into The Hope Buss dashboard.

2.     User navigates to the Stories & Updates section.

3.     User clicks “Create New Post” at the top of the page.

4.     A new story template opens, allowing the user to enter a title, content, and images.

5.     User reviews the content and clicks “Publish” to post it to the public website.

 

SubFlows:

Alternate/Exceptional Flows:

 

A1. Schedule → system publishes at scheduled time; can unpublish/rollback.

 

 

 

 

 

 

2.1.C. Use Case 16: View Testimonials

 

Use Case Name: View Testimonials

ID: 16

Importance Level: Low

Primary Actor: Vistitor

Use Case Type: Detail, Essential

Stakeholders and Interests:

 

-    Visitor: wants to read real stories and testimonials to learn how The Hope Buss impacts others.

-    Administrator: wants to ensure all testimonials are appropriate, accurate, and reflect the organization’s values.

Editor: wants to feature meaningful testimonials that inspire donors and volunteers to get involved.

Brief Description: Visitors browse testimonials and success stories.

Trigger: Triggered by Use Case 15: Publish Stories & Updates (new testimonials may be added or linked when a story is published).

Type: Internal

 

Trigger: Visitor or Community Member clicks on the “Testimonials” or “Community Stories” section on The Hope Buss website.

Type: External

Relationships:

Association: Administrator, Visitor, Community Member

Include: UC15 – Publish Stories & Updates (since testimonials may be included within published stories)

Extend: UC17 – Post Comments on Stories (visitors can leave feedback or comments on testimonials)

Generalization:

Normal Flow of Events:

 

1.     Visitor or Community Member navigates to The Hope Buss website.

2.     The user clicks on the “Testimonials” or “Community Stories” tab from the main menu.

3.     The system loads a page displaying testimonials from volunteers, donors, and families served.

4.     The user scrolls through and reads individual testimonials or stories.

 

SubFlows:

Alternate/Exceptional Flows:

 

1a. No testimonials are currently available:
 – The system displays a message: “No testimonials found. Please check back later.”

2a. If the page fails to load due to a network issue:
 – The system shows an error message: “Unable to load testimonials. Please refresh or try again later.”

3a. If the user clicks a broken or missing testimonial link:
 – The system redirects to the main testimonials page and displays: “This testimonial is no longer available.”

3. Class Diagram

 

 

 

 

Many internal impact dashboards are available to staff and administrators, and many of these can be opened by many staff. Each one of these has live data and charts summarizing meals served, families reached, volunteer hours, and donations. Visitors can see public impact counters that give a simplified version of similar information. Administrators can create, edit and delete stories or updates on the website. These are community work and volunteer efforts that are featured on the site and viewable by visitors. Each story has several testimonials, and each testimonial relates back to one story. The visitors can browse, read stories and testimonials, comment on them, while the administrators moderate and approve comments where needed. In addition, internal data export by administrators or staff is possible for tracking long-term community growth. In other words, the administrators of The Hope Buss manage data and content across dashboards, stories, and testimonials, while visitors engage with the published updates showcasing the impact of the organization.
